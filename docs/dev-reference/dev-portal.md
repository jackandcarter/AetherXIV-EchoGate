# Aether Dev Portal

The local dev portal is served by `tools/run-web.sh` and is available at:

```text
http://127.0.0.1:8080/dev/
```

The first portal module is the Enemy Restoration Workbench. It reads the local
server database and shows:

- decoded client CSV import and server/database diff results
- actor class, appearance, pool, group, and spawn readiness
- battle NPC presentation readiness (`propertyFlags` and event conditions)
- battle NPC skill/spell list IDs and row counts
- restoration evidence counts and statuses from `server_battlenpc_restoration_evidence`
- missing or present base Lua scripts for actor class paths
- provisional `!pinspawn` groups
- appearance review state from `server_battlenpc_appearance_audit`
- recent diagnostic trace files from `/tmp/aetherxiv-traces`
- safe in-game preview commands

## Client Decode Import / Diff

Use this section when you have decoded 1.23b client CSVs, especially:

```text
actorclass.csv
actorclass_graphic.csv
xtx_displayName.csv
```

Generate those CSVs from the launcher-selected local client with:

```bash
./tools/client-decode-csv.sh --include-display-names
```

By default the helper reads Echo Gate's saved `ClientRootPath`, then exports to:

```text
Data/client_exports/ffxivtool/decode_csv/
```

The generated CSVs are client-derived local data and are ignored by git.

The importer writes to local staging tables:

- `client_decode_import_batches`
- `client_decoded_actor_class_stage`
- `client_decoded_actor_graphic_stage`
- `client_decoded_display_name_stage`

The known old generator used actor ID column `0`, display name column `6`, and
graphic fields beginning at column `7`. The display-name dictionary uses ID
column `0`, singular name column `2`, and plural name column `3`. The class
path column was not preserved in the old helper, and the decoded `actorclass.csv` is only an
actorClassId-to-displayNameId spine. The portal defaults class path to `-1`;
for the CSVs generated by `tools/client-decode-csv.sh`, this stays blank and
monster/NPC filtering falls back to actor ID prefixes and graphic base model
families.

Known actor-class sheet IDs used by the helper:

- `actorclass`: `0x01030008`
- `actorclass_graphic`: `0x01030042`
- `xtx/displayName`: `0x0B450000`

After import, use the diff table to find rows where the client data says an
actor/appearance exists but the server database is missing class paths,
appearance rows, battle NPC pools, groups, or spawn rows.

The diff table also mines class-path leads from existing server data. A
`path lead` means another nonblank actor class already matches the same client
name and model base. A `base lead` means only the model base matches, so it is
weaker family evidence. `no path lead` means the current local data does not yet
prove a class path for that actor family; do not silently promote those rows.

Client decode rows prove identity/model linkage better than screenshots do, but
they still do not prove overworld placement. Use packet captures, archival
media, or `!pinspawn` review work for coordinates and density.

## In-Game Preview

Use `!previewappearance <appearanceId>` for incomplete enemy candidates. It
spawns a known-good shell actor and applies the candidate appearance, so blank
or unverified `classPath` rows do not need to be guessed. The preview actor is
labeled as `app <appearanceId>` when the running map server includes the
`SetCustomDisplayName` helper.

Use `!previewrange <startAppearanceId> <count>` to lay out a small labeled grid
of nearby appearance IDs. This is the fastest way to visually search a local
range when the database suggests the right area but the exact actor is unclear.

Use `!previewpair <serverActorClassId> <appearanceId>` when a row has enough
server data to attempt a real spawn. It places the real actor class beside the
safe-shell appearance preview. Mark the portal row as `Match`, `Wrong`,
`Unsafe`, or `Unsure` after comparing them in game.

Use `!spawn <actorClassId>` only for actor classes with nonblank, verified class
paths.

The portal audit table is intentionally separate from durable spawn tables.
Rows in `server_battlenpc_appearance_audit` are workbench evidence only; promote
confirmed enemy data through explicit SQL migrations.

## Restoration Matrix

The Enemy Readiness table is stricter than a visual preview. A row is only
`Restorable` when it has:

- a nonblank actor class path
- appearance data
- a resolvable base Lua script
- battle NPC presentation flags and event conditions
- pool, group, and spawn rows

Skill and spell lists may be empty for auto-attack-only provisional enemies. Add
ability rows only when client data, server traces, or historical sources support
the command IDs.
